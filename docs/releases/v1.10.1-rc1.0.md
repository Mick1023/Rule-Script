# RuleScript v1.10.1-rc1.0

RuleScript v1.10.1-rc1.0 is a release candidate for host-driven trigger execution. It adds HostTrigger metadata for script functions, a long-running runtime API, and `trigger task` dispatch inside `parallel` blocks.

## HostTrigger Functions

Script functions can be marked as host triggers with `@HostTrigger("Name")`:

```rulescript
@HostTrigger("LotArrived")
function OnLotArrived(lotId: string) -> void:
    Log("received " + lotId);
    return;
endfunction
```

The trigger name is separate from the function name. Hosts enqueue by trigger name, while RuleScript executes the marked user function.

## Trigger Task Dispatch

A `trigger task` must live directly inside a `parallel` block. The task waits for host trigger requests and dispatches one request each time it executes `dispatch;`.

```rulescript
parallel:
    task:
        LifeBit();
    endtask

    trigger task:
        dispatch;
    endtask
endparallel
```

The host thread does not directly call the user function. Requests are queued and consumed by the interpreter-owned trigger task.

## Runtime API Example

Create a runtime, inspect available triggers, start execution, and enqueue trigger requests:

```csharp
using RuleScript.Core.Runtime;

var engine = new RuleScriptEngine
{
    ExecutionTimeoutEnabled = false
};

engine.RegisterFunction(
    "Log",
    [new RuleScriptParameterSymbol("message", RuleScriptValueType.String)],
    RuleScriptValueType.Null,
    args =>
    {
        Console.WriteLine(args[0]);
        return null;
    },
    threadSafe: true);

var runtime = engine.CreateRuntime("""
    @HostTrigger("LotArrived")
    function OnLotArrived(lotId: string) -> void:
        Log("lot arrived: " + lotId);
        return;
    endfunction

    parallel:
        trigger task:
            dispatch;
        endtask
    endparallel
    """);

foreach (var trigger in runtime.HostTriggers)
{
    Console.WriteLine($"{trigger.HostTriggerMetadata?.Name}: {trigger.Signature}");
}

var running = runtime.StartAsync();

await runtime.TriggerAsync("LotArrived", ["LOT-001"]);
await runtime.TriggerAsync("LotArrived", ["LOT-002"]);

await runtime.StopAsync();
await running;
```

## Queue Behavior

- `TriggerAsync(name, args)` enqueues a request and returns a receipt.
- Requests are processed FIFO.
- Repeated requests with the same trigger name are preserved and processed separately.
- Unknown trigger names are rejected before enqueue.
- Unmarked functions cannot be triggered by the host.
- If the runtime has no active `trigger task`, trigger enqueue fails fast instead of creating work that can never be consumed.

## Metadata

Host trigger metadata is exposed through `RuleScriptFunctionSymbol`:

```csharp
var runtime = engine.CreateRuntime(script);

foreach (var trigger in runtime.HostTriggers)
{
    Console.WriteLine(trigger.Name);
    Console.WriteLine(trigger.HostTriggerMetadata?.Name);
    Console.WriteLine(trigger.ReturnType);
    Console.WriteLine(string.Join(", ", trigger.Parameters.Select(p => $"{p.Name}: {p.Type}")));
}
```

The analysis API also exposes marked functions through `RuleScriptAnalysisResult.HostTriggers`.

## Compatibility Notes

- Existing `function`, `task`, and `parallel` behavior is preserved.
- `@` is now a valid token for function attributes.
- `trigger` is now a reserved keyword.
- `trigger task` currently requires asynchronous runtime execution.
- This is a release candidate API and may receive small naming or lifecycle refinements before stable release.

## Validation

The v1.10.1-rc1.0 work is covered by tests for:

- HostTrigger tokenization
- HostTrigger function metadata parsing
- `trigger task` parsing inside `parallel`
- rejecting `trigger task` outside `parallel`
- exposing HostTrigger metadata through analysis
- exposing HostTrigger metadata through runtime
- FIFO host trigger request processing
- repeated same-name trigger enqueue
- rejecting unmarked function trigger attempts
- full existing regression coverage
